01 概述与背景

1.1 Flutter 的镜像需求

Flutter 是 Google 开源的跨平台 UI 框架，其开发工具链依赖多个 Google 托管资源：SDK 引擎二进制、Dart 包管理器（pub.dev）、以及 SDK 源码仓库（GitHub）。在中国大陆网络环境下，三个环节都可能受阻：

  • flutter upgrade / flutter create 时从 storage.googleapis.com 下载引擎构建产物失败
  • flutter pub get 从 pub.dev 拉取依赖包超时
  • flutter upgrade 从 GitHub 拉取 SDK 源码 git pull 失败

Flutter 的镜像配置涉及 三个独立维度，需要作为一个整体管理。

注意：Flutter 镜像不同于 Node.js（单环境变量）或 ohmyzsh（单 git remote）。它是一个 多维度绑定配置——每个镜像源同时提供 FLUTTER_STORAGE_BASE_URL、PUB_HOSTED_URL 和可选的 SDK git remote。

1.2 三个配置维度

维度配置方式影响范围对应现有模式
Flutter Infra（引擎二进制） 环境变量 FLUTTER_STORAGE_BASE_URL flutter upgrade、flutter create 时下载引擎产物 shell profile 写入（同 Node.js）
Dart Pub（包 registry） 环境变量 PUB_HOSTED_URL flutter pub get 拉取依赖包 shell profile 写入（同 Node.js）
Flutter SDK 源码仓库 git remote set-url origin flutter upgrade 的 git pull 步骤 git remote 操作（同 ohmyzsh）
本次排除：download.flutter.io（Android 构建时 Gradle 下载 Java 依赖）需要修改 build.gradle，配置复杂度高且非常见场景，建议后续版本单独设计。

2.3 关键交互决策

决策点方案理由
是否管理 SDK git remote 是 — 仅对提供 SDK 镜像的源生效 TUNA 和 SJTUG 提供了 flutter-sdk.git 镜像；CFUG/腾讯云不提供则跳过 git remote 步骤
切换确认弹窗 需要确认 — 同 Brew/GitHub Flutter 配置涉及 shell profile 写入（影响新终端），需要用户确认
测速策略 HEAD 请求 FLUTTER_STORAGE_BASE_URL Storage 地址是 Flutter 的核心依赖，最能反映实际体验
SDK 安装检测 检测 flutter 命令是否在 PATH 中 与 ohmyzsh 检测 ~/.oh-my-zsh 类似，确保 flutter 已安装
SDK git remote 操作条件 仅当镜像提供 SDK remote 且 flutter SDK 是 git 仓库时执行 某些用户通过 tarball 安装 Flutter（非 git clone），此时不支持 remote 操作

03 技术方案

3.1 架构定位

Flutter 是 MirrorKit 中第一个 多维度配置 的镜像类型。它的实现结合了两种现有模式：

配置维度实现机制参考模式
环境变量写入 shell profileShellProfileManager.writeBlock()Node.js
git remote set-urlCommandExecutor.run("git", ...)ohmyzsh
Flutter 安装检测which flutter 或检测路径ohmyzsh（类比）

05 设计决策记录

#决策选择替代方案理由
D1 配置组织方式 1 个入口 = 6 个镜像源，每个源是一个配置包（storage + pub + sdk） 拆成 Flutter Infra / Dart Pub / Flutter SDK 三个独立页面 三个维度强耦合——用户切换镜像站时三个值需要一起变。拆开会导致配置不一致（如 storage 用清华但 pub 用 CFUG）
D2 SDK git remote 处理 有条件执行（仅在镜像提供 remote 且 SDK 是 git repo 时） 总是执行 / 永远不执行 CFUG/腾讯云不提供 SDK 镜像则跳过；tarball 安装用户没有 git remote
D3 切换确认弹窗 需要 confirmDialog（同 Brew/GitHub） 无确认直接切换（同 Node/PyPI） 涉及 shell profile 写入，影响所有新终端会话。用户应被告知"修改将在新终端生效"
D4 测速目标 HEAD 请求 FLUTTER_STORAGE_BASE_URL 请求 PUB_HOSTED_URL 或两者都测 storage 地址是 Flutter 最核心的依赖（引擎二进制 ~200MB），最能代表实际体验
D5 未安装引导 自动检测 flutter 命令，未安装时显示引导页 不检测，直接显示管理界面 与 ohmyzsh 行为一致——确保用户在已安装 Flutter 后才有意义
D6 排除 download.flutter.io 本次不做 一并实现 需要修改 Gradle 构建配置，范围超出"镜像切换"。属于 Android 构建调试范畴，后续版本单独设计

06 风险与应对

风险等级应对
FLUTTER_STORAGE_BASE_URL 路径结构不一致（各镜像站路径不同） 中 在模型中为每个镜像硬编码完整 URL，不依赖工具自动拼接 /flutter_infra
用户通过 tarball 安装 Flutter（非 git clone） 低 先检测 flutter 所在目录是否为 git 仓库，不是则跳过 git remote 操作
pub.dev 镜像仅支持 Dart 2.8+ 极低 当前 Flutter 最低要求 Dart 3.x，兼容性无问题
腾讯云 CNIC 镜像数据滞后 低 测速结果会反映延迟和可用性；在列表中以非推荐标识
